IF YOU ARE TO PREPARE A ROUND ON CODEFORCES, THESE ADVICES ARE RULES. FOLLOW THEM AS CLOSE AS YOU CAN.

* are for Codeforces only.

Checker, validator and interactor

  • Use testlib.h. Read help here.
  • Do not use defines from your template. Write understandable code.

Checker:

  • Use std::ncmp.cpp unless you really need another checker.
  • Use a standard checker if possible.
  • Use std::yesno.cpp for yes/no problems.
  • In custom checkers do not check whitespaces unless you really need to (don’t use readLine(), use readToken() if possible).
  • Do not validate input file in checker (including the use of inf.readSpace()/inf.readEoln()).
  • Always validate output (both contestant's and jury's), like answer = readInt(1, n, “answer”).
  • Always read and check contestant's and jury's answer in the same way (use readAns paradigm). Do not forget to initialize all global variables in the beginning of readAns.
  • * Checker comment on examples is visible to contestants. So:
  • Do not spoil anything in the comments.
  • Make checker’s comments easy to understand.

Tests and generators

  • Do not use the "add tests from archive/files" feature. Upload manually only tests that you printed by hand, upload generators for others.
  • Do not use the following syntax in polygon scripts: gen > {1-100}. Use generators that generate exactly one test.
  • Make distinguishable names for generators. Bad names: gen1, gen2, gen3. Bad names (passed as a parameter): gen type=1, gen type=2, gen type=3. Good names: gen_path, gen_binary, gen type=random.
  • Always add all possible small tests (like with n = 1, 2, 3) if the total number of them is reasonable.
  • There are 20 to 100 tests usually.
  • Always make tests (at least) with minimum possible, maximum possible and some value in between for each variable and their combinations.
  • * In general, pretests should include: a test with maximum size of input (e.g. maximum n), a test against integer overflow, a few random small tests. All corner cases that are easy to miss. If you deliberately don’t include some corner case, talk to the coordinator about this.
  • * In D2A and D2B use test cases. There should be 2-3 pretests for D2A and no more than 3-4 pretests in D2B.
  • * Pretests number should be between 7-9 for D1A and up to 10-20 for D1E.
  • * At least one example for each possible output format. At least 2 examples/test cases in total, it’s better to have more examples in easier problems.
  • Example explanations are a must.
  • Always create at least one stress on random big tests with correct solutions: other correct solutions (e.g. tester’s) will be added to it and checked. Also create a stress with random small tests.
  • Do not use a generator with the same parameters (but different seeds) more than once.

Statements

  • Do not make extremely long statements. The legend should fit into the initial size of the textarea in polygon.
  • Use spell checkers (and grammar checkers if you are unsure in your English).
  • Words in English titles should almost always start with capital letter: https://capitalizemytitle.com
  • Use common phrases and parts in order to write usual things. If you have already seen a problem that has the same sentence you want to add, go and copy it.
  • If some of the following items could be used in your statements, copy them.

\begin{itemize}

\item in the first position where $a$ and $b$ differ, the string $a$ has a letter that appears earlier in the alphabet than the corresponding letter in $b$.

\end{itemize}

A permutation is an array consisting of $n$ distinct integers from $1$ to $n$ in arbitrary order. For example, $[2,3,1,5,4]$ is a permutation, but $[1,2,2]$ is not a permutation ($2$ appears twice in the array) and $[1,3,4]$ is also not a permutation ($n=3$ but there is $4$ in the array).

After printing a query do not forget to output end of line and flush the output. Otherwise, you will get \t{Idleness limit exceeded}. To do this, use:

\begin{itemize}

\item \t{fflush(stdout)} or \t{cout.flush()} in C++;

\item \t{System.out.flush()} in Java;

\item \t{flush(output)} in Pascal;

\item \t{stdout.flush()} in Python;

\item see documentation for other languages.

\end{itemize}

Answer ``\t{***}'' instead of ``\t{***}'' or ``\t{***}'' means that you made an invalid query. Exit immediately after receiving ``\t{***}'' and you will see \t{Wrong answer} verdict. Otherwise you can get an arbitrary verdict because your solution will continue to read from a closed stream.

Smth is as follows. Always singular!

  • Give constraints on each input value in parentheses right after you said where it is contained. The parentheses should not be included in the TeX formula. Example: $n$ ($1 \le n \le 10^5$).
  • In input section, always first list all numbers on a line, then give constraints. Wrong: two integers $n$ ($1 \le n \le 100$) and $m$ ($1 \le m \le 100$). Right: two integers $n$ and $m$ ($1 \le n, m \le 100$).
  • Use only lowercase letters for variables. ($n$, not $N$).
  • All text elements (words, spaces, punctuation marks) should not be included in TeX formulas. Wrong: two integers $x, y$. Right: two integers $x$, $y$.
  • All variables and important constants should be in TeX formulas. Wrong: m does not exceed one hundred. Right: $m$ does not exceed $40$.
  • In input section, make the structure of text reflect the structure of input. If possible, describe in different paragraphs input that is given in different lines, and in one paragraph input that is given in one line.
  • For HTML statements (including Codeforces) use \t{} to write constants starting with unary minus. (deprecated)
  • To denote a sequence, use only its id or list all elements. Wrong: a sequence $x_i$. Right: a sequence $x$ of length $n$, or, even better, a sequence $x_1, x_2, \ldots, x_n$.
  • Write statements in TeX:
  • Write all formulas in dollars.
  • For dash use ~--- in English and html Russian, “--- in pdf-only Russian. Example: word1~--- word2, word1 “--- word2
  • For indices use $i$-th and similar ($i$-й, $i$-я, $i$-го in Russian).
  • Use double line break to start a new paragraph.
  • Use italic \textit{} to introduce a definition. Use bold \textbf{} to mark something important. Use them only for a word or several words. If you want to bold a sentence or a few, rewrite the statement.
  • Use \cdot in formulas for multiplication.
  • Use \bmod for modulus operation. Like In GCD algorithm, you repeatedly replace $(a, b)$ with $(b, a \bmod b)$.
  • Use \equiv and \pmod to show equivalence modulo some number. Example: $a^{p - 1} \equiv 1 \pmod{p}$ holds for any prime $p$. Feel the difference from the previous option!
  • Write clear text. End statements with dots. Start statements with capital letters. Place spaces properly. Wrong: `word1,word2`, `word1(word2)word3`, `word1 ( word2 ) word3`. Right: `word1, word2`, `word1 (word2) word3`.
  • Do not use the phrase “space-separated” unless it is really needed.
  • Use square brackets to denote sequences and arrays (array is $[3, 4, 5, 3]$) and curly brackets to denote sets and multisets with no general order (the good values are $\{3, 4, 7\}$).

Solutions

  • For each problem there should be:
  • Main correct solution -- written without the usage of non-asymptotic optimizations. It should use at most half TL and half ML.
  • If there is a possibility of Java solutions working slow (including, but not limited to C++ solutions working more than a ¼ of TL or C++ solutions using more than 50 MB), a correct java solution is needed.
  • Other correct solutions, including at least one by another author. All correct solutions should fit half TL and half ML.
  • A stupid bruteforce solution that does exactly what is written in the statements. Make sure it only gives OK or TL (place while (true); if needed). Select tag Time Limit Exceeded for it.
  • Slow solutions that you do not want to pass. Optimize them as much as you can, including using pragmas. These solutions should not fit double TL or ML.
  • For easy problems write a solution in Python and make sure it passes without any optimizations.
  • Do not use g++64 in the main correct solution.
  • If there is a possibility of overflow in the main solution, write one in Python.
  • If there is a possibility of incorrect solutions due to overflow, add one for each variable/array/place in code where overflow can occur. Make sure they all fail on pretests.
  • If possible, do not make tight limits. For harder problems, 2 seconds TL is minimum possible.
  • If there is a solution with case work, or some part of the problem requires case work, write one. Then add a wrong solution for each case commented out. Make sure they all fail on pretests.

General

  • * There should be no warnings in polygon, except (if reasonable) "there are tests with input/output for statements [without TeX marks]".
  • Tag Time Limit Exceeded or Correct can be used only for solutions on the bound of TL. For solutions that get TL, use tag Time Limit Exceeded
  • Editorials should be written in the Statements tab, under the statement.
  • * For Codeforces round editorials in English should be ready with the end of the round.
